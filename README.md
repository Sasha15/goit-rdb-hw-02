p0_initial_schema - показує схему первинної бази даних до виконання нормалізації
p2_split_product_name_and_quantity - розділяємо атрибут split_product_name_and_product_quantity на два окремих атрибута: product_name та product_quantity для нормальної форми 1. Після цього, номер замовлення стає складним і атрибут user залежить від частини номера замовлення а не всього ключа, так як user "Мельник" замовив 3 штуки лептопів і 2 мишки, то після першої нормалізації в таблиці буде два записи з ключем 101.
p3_split_orders_and_product_details - щоб в таблиці не було повторень виносимо товари в окрему таблицю "Товари в замовленнях" (orders_products). І встановлюємо звязки один до багатьох. Тобто одне замовлення може мати багато продуктів.
p4_move_user_to_separate_table - виніс всі дані по клієнту в окрему таблицю users, також розділив адресу на поля для досягнення атомарності і привести її до НР1. Звязок один до багатьої. Один клієнт може мати багато замовлень.
p5_ready_ER_diagram - готова діаграма з залежностями. Додатково виніс продукти з табилці "товари в замовленнях" в окрему таблицю. Залежність один товар до багатьох продуктів в замовленнях.
